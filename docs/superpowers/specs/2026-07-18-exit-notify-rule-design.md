# 大口の売り抜け通知(5%割れ)設計書

- 作成日: 2026-07-18
- ステータス: ユーザー承認済み(実装前)

## 目的

変更報告書で大口保有者の株券等保有割合が**前回5%以上 → 今回5%未満**になったもの
(=大口の事実上の撤退。需給悪化シグナル)をDiscordに通知する。

実データでの裏付け(2026-07-08〜07-11の291件): 5%割れは23件(1日7〜8件)。
訂正の重複を除くと1日5件前後の通知量になる見込み。例: ディー・エヌ・エー[2432]に対する
シティインデックスイレブンスの5.06%→3.76%など、狙いどおりのシグナルが含まれる。

## 方式(検討3案から案Aを採用)

- **案A(採用)**: rules.ymlの`match`に数値条件の語彙を追加。通知追加が今後も設定編集だけで済む
- 案B(不採用): collector側で専用カテゴリ`exit`を付与 — 「カテゴリ=書類種別」の原則が崩れ、閾値がコードに埋まる
- 案C(不採用): notify.pyに直書き — 拡張性なし

## 変更内容

### 1. `collector/notify.py`

- `match`の数値条件を2つ追加:
  - `prev_ratio_min: <数値>` — `prev_ratio`が数値以上ならマッチ
  - `ratio_below: <数値>` — `ratio`が数値未満ならマッチ
  - **`ratio`/`prev_ratio`がNoneの書類は条件不成立(通知しない)**。フェイルセーフは
    取りこぼし側に倒す(タイムラインには残るため後から確認可能)
- 判定関数を「マッチしたか(bool)」から「**最初にマッチしたルール(dict) or None**」を返す形に変更
- embed生成時、マッチしたルールに`label`/`emoji`があればカテゴリ既定の見出しを上書き。
  売り抜けルールは赤系(0xD32F2F)の専用色

### 2. `config/rules.yml`

```yaml
  - name: 大口の売り抜け(5%割れ)
    label: 売り抜け
    emoji: "📉"
    match:
      category: [change_report]
      title_not_contains: [訂正]
      prev_ratio_min: 5
      ratio_below: 5
```

既存の「新規の大量保有報告書」ルールは無変更。

### 3. `collector/run.py`

判定関数の戻り値変更(bool→ルール)に追従し、通知時に(アイテム, マッチしたルール)を渡す。

### 4. `docs/DESIGN.md` §6.1

matchの語彙(`prev_ratio_min` / `ratio_below`)とルールの表示設定(`label` / `emoji`)の説明を追記。

## 変更しないもの

- collectorの収集・保有割合パース・JSONLスキーマ・GitHub Actionsワークフロー
- 既存の「新規5%超え」通知の挙動

## 通知例

```
📉 売り抜け | 変更報告書(株式会社シティインデックスイレブンス)
**[2432] ディー・エヌ・エー**
提出者: 株式会社シティインデックスイレブンス
保有割合: 3.76% (前回 5.06%)
提出時刻: 15:02
```

## テスト計画

1. モックで境界値検証: 5.0%ちょうど(prev=5はマッチ、ratio=5は不マッチ)、
   ratio/prev None → 不成立、訂正タイトル → 除外、既存ルールへの非干渉
2. 蓄積済み実データ(291件超)へのドライラン: 「稼働していたら通知されていた一覧」を出力し
   件数・内容が想定(訂正除外後の5%割れ)と一致することを確認
3. Discordへテスト通知1件を実送信し、📉表示・赤色・割合表記を目視確認
4. Python 3.7(ローカルvenv)で全モジュール構文チェック

## 受け入れ条件

- 5%割れの変更報告書が📉「売り抜け」としてDiscordに届く
- 訂正報告書・保有割合が取れない書類では通知が発生しない
- 既存の新規5%超え通知が従来どおり動く
